## 1. 概要・目的

### 1.1 背景
福利厚生サービス「リアライズクラブ（RC）」での企業導入を入口に、**完全予約制でマイホーム提案へつなぐ B2B2C 導線**（賃貸窓口 → マイホーム）を支える営業支援ツール。一般的な不動産営業ではなく、「人生相談から住居費設計へ」視点を移す**トップクローザー育成型の営業プロセス**をシステム化することを目的とする。

### 1.2 目的
- 反響〜契約までの商談プロセス（反響・予約 → 人生相談 → 4章プレゼン → 交渉 → 契約）を1ツールで可視化・管理する
- 面談音声を**AIで自動文字起こし・議事録化**し、入力作業をゼロに近づける
- 「人生相談カルテ」で顧客の不安を構造化し、AIが提案トーク（切り口）を生成する
- クローザーの育成（標準カリキュラムと習得度管理）を仕組み化する

### 1.3 設計方針
- **自己完結 + 段階拡張**：APIキー無し（デモモード）でも全機能が動作し、キーを入れると本物のAIに切り替わる
- **1コンテナ完結**：FastAPI 1プロセス + SQLite で、ローカルでもクラウドでも同一コードで動く
- **データはユーザー管理下**：ローカルはPC内、クラウドは永続ディスク（`/data`）に保存

## 5. データベース設計

SQLite 単一ファイル（`sales_tool.db`）。全テーブルが顧客（companies）・商談（deals）を軸に連動。外部キーは `ON DELETE SET NULL`（育成進捗のみ CASCADE）。

### 5.1 テーブル一覧

| テーブル | 役割 |
|----------|------|
| `companies` | 顧客企業（CRM中核） |
| `contacts` | 担当者（会社にひもづく） |
| `deals` | 商談／パイプライン（`presentation` に4章プレゼン進捗JSON） |
| `tasks` | タスク（抽出・ToDo） |
| `meetings` | 面談 ＋ **AI議事録**（後付けAIカラム群を同居） |
| `hearing_sheets` | 人生相談カルテ（`talk_points` にAIトークJSON） |
| `training_modules` | 育成カリキュラム |
| `training_progress` | メンバー×モジュールの習得進捗（1組1行 upsert） |

### 5.2 主要カラム（抜粋）

**deals**：`company_id, title, stage, amount, probability, expected_close, owner, notes, presentation(JSON), created_at, updated_at`

**meetings（AI議事録カラムを含む）**：
基本：`company_id, contact_id, deal_id, title, scheduled_at, duration_min, location, meeting_type, status, agenda, minutes`
AI拡張：`audio_filename, diarize, transcript, summary, minutes_md, dialogue_md, utterances(JSON), speaker_map(JSON), decisions(JSON), next_actions(JSON), tags(JSON), ai_status, error_message`

**hearing_sheets**：`current_situation（家計）, challenges（住環境）, needs（万が一）, budget（老後）, authority（災害）, timeline（健康）, competitors（想い・背景）, next_action（次の一手）, raw_voice_text, talk_points(JSON)`
※カラム名はBANT由来の汎用名のまま、画面上は人生6大項目として表示。

### 5.3 マイグレーション設計
- 起動時 `init_db()` → `_migrate()` で**不足カラムを自動 ALTER**（既存DBを壊さず拡張）
- 旧ステージ（リード/アプローチ/ヒアリング/提案/見積/クロージング/受注）を**新ステージへ自動移行**（`STAGE_MIGRATION`）
- 育成カリキュラムを**トップクローザー版へ自動差し替え**（新版未投入時のみ）

## 6. ドメインロジック

### 6.1 商談ステージ（`DEAL_STAGES`）
```
反響・予約 → 人生相談 → プレゼン → 交渉 → 契約 ／ 失注
```
- `契約` へ移動で確度を自動100%、`失注` で0%に設定
- 不正なステージ値はAPI側で400エラーとして拒否（集計エラーの根本防止）

### 6.2 4章プレゼン進捗（`PRESENTATION_STEPS`）
```
人生相談 → ①家は買うべき → ②今買うべき → ③ここを買うべき
→ ④うちから買うべき → 資金計画 → クロージング
```
商談ごとにチェックリスト（JSON）として保存し、クロージング導線の進捗を可視化。

### 6.3 育成カリキュラム
「トップクローザー研修プログラム」準拠の10モジュール（基礎／人生相談／プレゼン／交渉・クロージング／契約・手続き／集客・RC導線／ツール活用）。AI先生（ミオ／アオ／ディグラムGPT）やリアプラ活用も学習項目に含む。

## 8. AI議事録パイプライン設計

### 8.1 処理フロー（非同期）
```
[POST /api/minutes/upload]
  1. 拡張子検証（mp3/m4a/wav/webm/mp4/ogg/flac/opus 等）＋サイズ上限（既定500MB）
  2. UUIDファイル名で data/uploads へ保存、meetings 行を ai_status="queued" で作成
  3. BackgroundTasks で process_minutes() を起動（プロセス内）
        ↓
  4. processing → 文字起こし
       ├ 話者分離ON & AssemblyAIキー … diarize（話者ラベル付き）
       ├ LOCAL_WHISPER=true        … ローカル faster-whisper（無料・キー不要）
       └ 既定                       … OpenAI Whisper API
        ↓ transcribed
  5. summarizing → Claude で議事録一式生成（要約／議事録MD／決定事項／次アクション／タグ）
        ↓ summarized（完了）  ※失敗時は error + error_message
```
- フロントは2.5秒間隔のポーリングでステータスを監視（最大約25分）
- `POST /api/minutes/text`（貼り付け）と `POST /api/minutes/{id}/generate`（生成）は同期処理

### 8.2 状態管理・障害復旧
- 状態は `meetings.ai_status` 列で管理：`queued → processing → transcribed → summarizing → summarized`（失敗 `error`）
- 専用ジョブテーブルは持たず `meetings` に同居（`jobs_sql.py` が操作）
- 再起動でプロセス内タスクが消えるため、起動時 `recover_stuck()` が中断ジョブを `error` に倒す。`reprocess` で再キュー可能

### 8.3 連携
- 議事録の**次アクション → タスク化**（全件 `actions_to_tasks` ／単件 `action_to_task`）
- 議事録を**会社・商談に紐付け**（`link`）、Markdownエクスポート（`export.md`）

## 9. 認証・セキュリティ設計

| 項目 | 内容 |
|------|------|
| 認証方式 | 環境変数 `APP_PASSWORD` による単一パスワード認証（未設定なら無効＝全公開） |
| セッション | Cookie `lms_session`（httponly / samesite=lax / 7日）。トークンは `sha256("lms:"+APP_PASSWORD)` |
| 保護範囲 | ミドルウェアで `/api/*` を保護。除外は `/api/auth/*` と `/api/ai/health` のみ。フロント本体（`/`, `/static`）は無認証配信 |
| セキュリティヘッダ | 全レスポンスに `X-Content-Type-Options: nosniff`、`X-Frame-Options: DENY` |
| キャッシュ制御 | `/` と `/static` に `no-cache, must-revalidate`（更新後の古いJS/CSS残留防止） |
| 入力検証 | ステージ検証、アップロードのサイズ／空ファイル／拡張子検証 |

> **運用上の注意**：公開時は `APP_PASSWORD` 必須。トークンは固定値のためCookie漏洩時の個別失効はできない（運用ではHTTPS必須・パスワード定期変更を推奨）。本格運用では多要素・ユーザー別認証への拡張余地あり。

## 12. 既知の課題・今後の拡張

### 12.1 既知の不具合（✅ 対応済み）
- ~~フロントの議事録削除処理（`delMinutes`）が `/api/minutes` ではなく `/api/meetings` を呼んでいる不整合~~
  → **修正済み**：専用の `DELETE /api/minutes/{id}` を新設し、議事録レコードに加え**残骸となるアップロード音声ファイルもディスクから削除**するようにした。フロントも同APIを呼ぶよう変更。

### 12.2 拡張ロードマップ

**✅ 実装済み（2026年6月）**
- **タスク抽出のAI化**：`ai_services/ai.py` の `extract_tasks_ai()` を追加。`task_extractor.extract_tasks()` は「APIキーあり＆DEMO_MODE無効」なら Claude、未設定・失敗時はルールベースへ自動フォールバック。
- **カルテ振り分けのAI化**：同様に `classify_hearing_ai()` を追加し、`hearing_parser.parse_hearing()` が AI／キーワードを自動切替。
- **認証強化**：ユーザー別アカウント（`users`）・ロール（admin/member）・**セッション失効**（`sessions` テーブル）を実装。パスワードは pbkdf2-sha256。`auth.py` 新設。後方互換として、ユーザー未登録かつ `APP_PASSWORD` 設定時は admin を自動作成。管理者はUIからユーザーの追加・削除・PW再設定・無効化（＝即ログイン失効）が可能。

- **ジョブ基盤分離（Redis不要のDBキュー）**：AI議事録を `BackgroundTasks` の即時発火から、**`ai_status='queued'` をDBキューとして常駐ワーカースレッドが1件ずつ逐次処理**する方式へ変更（`jobs_sql.py`：`claim_next/worker_loop/start_worker`）。追加インフラ不要で1コンテナ維持。再起動時は中断ジョブを `queued` に戻して自動再処理（耐障害性向上）。SQLite書込競合に備え `busy_timeout` を設定。検証：3件同時アップロード→全件 `summarized`。
- **外部CRM連携（HubSpot）**：`hubspot_sync.py` を新設（httpx）。会社/担当者/商談を HubSpot へ push（`hubspot_id` で重複作成を防止）、HubSpotの会社を pull。`HUBSPOT_TOKEN` 未設定なら自動でオフ（AIと同じゲート方式）。管理者はCRM画面の「🔗 HubSpot」から同期実行。**実接続の検証には Private App トークンが必要（未提供のため未検証）。**

**🔜 見送り**
- **データ基盤（PostgreSQL化）**：現状の規模では SQLite で十分のため見送り。全 INSERT が `lastrowid` 依存・プレースホルダ差異（`?`→`%s`）等があり、移行時は Postgres 実機での検証が必須。`DATABASE_PATH`/`DATABASE_URL` の切替余地は確保済み。
